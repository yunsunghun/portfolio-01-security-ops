# 포트폴리오 01 — 기획 브리프 (Portfolio Brief)

## 1. 한 줄 정의

**가용성·보안·운영 자동화를 동시에 만족하는 웹 서비스 운영 체계**를 코드와 문서로 재현·설명하는 저장소입니다.

## 2. 대상 독자

| 독자 | 이 저장소에서 보고 싶은 것 |
|------|---------------------------|
| **인사/채용** | 역할 정의, 협업 방식, 문서화 습관, 책임 범위의 명확성 |
| **엔지니어** | 아키텍처 판단, IaC·스크립트 품질, 장애 대응·보안 거버넌스의 깊이 |

## 3. 가상 시나리오 (프로젝트 설정)

- **서비스**: B2B API 게이트웨이 뒤의 내부 마이크로서비스 일부를 단순화한 **데모 웹 프론트 + 헬스 API**.
- **규모 가정**: 단일 리전, 다중 AZ는 문서로 확장 경로만 제시 (실제 클라우드 계정 불필요).
- **팀 가정**: SRE 1, 보안/관제 1, 백엔드 2 — **온콜·에스컬레이션**은 런북에 정의.

## 4. 범위 (In Scope)

1. **서버 운영**: 헬스체크, 디스크·로그 용량, 백업 절차(스텁 스크립트 + 체크리스트).
2. **서비스 운영**: 배포 전후 점검, 롤백 판단 기준, 데모 스택으로 트래픽 경로 설명.
3. **보안 관제**: 자산·접근 통제 개요, 취약점 대응 템플릿, CI 정적 점검(ShellCheck 등).

## 5. 비범위 (Out of Scope)

- 실제 프로덕션 자격 증명, 실고객 데이터.
- 특정 클라우드 벤더에 종속된 **필수** 배포 (대신 Terraform/Ansible **패턴**과 로컬 데모 제공).

## 6. 성공 지표 (포트폴리오 관점)

- 클론 후 **15분 이내**에 `README`만으로 데모 스택 기동 가능.
- **문서만 읽고도** 인시던트 시뮬레이션 1건을 역할별로 나눠 대응할 수 있음.
- 코드·워크플로가 **재현 가능**하고 린트/CI 통과를 전제로 함.

## 7. 02·03과의 역할 분리 제안

| 번호 | 제안 테마 |
|------|-----------|
| **01 (본 저장소)** | 보안관제 + 서비스·서버 운영 **통합** (런북·거버넌스·데모 스택) |
| **02** | 관측성 전문 (OpenTelemetry, 대시보드, SLO/에러 버짓) |
| **03** | 침해 대응/포렌식 또는 컴플라이언스(감사 로그, 정책 코드화) |

## 8. 유지보수 원칙

- 시연용 시크릿은 **환경 변수 + `.env.example`**만 사용.
- 모든 스크립트는 **POSIX 호환**을 목표로 하되 Windows Git Bash에서 동작 검증 권장.
