# 런북 (Runbooks)

## 공통: 인시던트 선언

1. **영향도**: 사용자 수, 매출/SLA 영향, 데이터 유출 여부(알려진 범위).
2. **심각도**: P1(전면 장애) ~ P4(경미/정보).
3. **Incident Commander**: 한 명 지정, 커뮤니케이션 채널 단일화.

---

## RB-01: 헬스체크 실패

### 증상

- `/health`가 비정상 HTTP 코드 또는 타임아웃.

### 즉시 조치

1. 최근 배포 여부 확인 → 있으면 **롤백 후보**로 표시.
2. 동일 호스트에서 `curl -sf http://127.0.0.1:8080/health` (데모 포트는 환경에 맞게 조정).
3. 컨테이너/프로세스 재시작 (데모: `docker compose restart`).

### 에스컬레이션

- 15분 내 복구 없음 → 애플리케이션 담당 + 인프라 담당 동시 소집.

---

## RB-02: 디스크 사용률 임계 초과

### 증상

- `automation/scripts/disk_check.sh`가 WARNING/CRITICAL.

### 조치

1. 로그·임시 파일 디렉터리 용량 확인 (`du -sh`).
2. 로그 로테이션 정책 적용 여부 확인.
3. 불필요 아티팩트 삭제(승인 하에).

---

## RB-03: 의심스러운 로그 패턴 (관제)

### 증상

- 동일 IP에서 짧은 시간 다수 4xx/5xx, 비정상 User-Agent.

### 조치

1. WAF/방화벽 규칙이 있다면 임시 차단 후보 목록 작성.
2. 애플리케이션 로그와 상관 → 버그 vs 공격 구분.
3. 보안 담당에게 **CVE/침해 가능성** 공유, 필요 시 포렌식 절차(별도 정책).

---

## 시뮬레이션 과제 (면접 대비)

**상황**: 배포 직후 헬스만 간헐 실패.  
**과제**: RB-01 순서대로 가정하고, 본인이 작성할 Slack/메신저 첫 메시지 3줄 + 다음 액션 2가지를 적어보기.
